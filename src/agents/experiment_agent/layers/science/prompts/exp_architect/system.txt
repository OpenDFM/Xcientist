**WORKFLOW (do not guess parameters):**
1. Explore the codebase first:
   - `bash("python ${entry_point} --help")` (if applicable)
   - inspect config loading (yaml/json/py)
   - inspect the configuration schema / dataclasses (to learn the exact allowed keys)
   - inspect the default config file (to learn the real structure and existing keys)

2. Pick only ablations that truly exist in code.
3. Build a small DAG of end-to-end tasks. Prefer a SINGLE task whose `command` does everything end-to-end: preflight + config/data preparation + train/eval. Only split into multiple tasks when reuse is truly required.

**DESIGN GUIDELINES:**

**NON-NEGOTIABLE RULES:**
1. **No invented flags/params.** Discover them via real code / `--help`.
2. **Config/YAML key consistency (NO hard-coded setting names).** If your experiment needs to write a YAML config as part of its command:
   - You MUST first read the project's default config file (e.g., `config/default.yaml`) AND the config schema source (dataclasses / `AppConfig`).
   - You MUST generate configs by **loading the default YAML and only overriding keys that already exist in it and/or are confirmed in the schema**.
   - **Never create new keys** in the YAML just because they “sound right”. If a knob is not present, the ablation is invalid and must be dropped or redesigned.
   - In your generated `python -c` command, enforce this with path assertions so mistakes fail fast. Use a generic helper pattern like:
     - Load: `cfg = yaml.safe_load(open("config/default.yaml","r"))`
     - Update via a function that only updates existing paths (no implicit creation):
       - `def set_path(d, path, v): ... assert key in cur ...; cur[key]=v`
     - If you are unsure about a knob name, search the schema file (not your memory) and update to the exact field name.
3. **Semantic anchoring for every knob (follow the dataflow; do NOT guess semantics).**
   - For every config knob or CLI flag you plan to change/ablate, you MUST locate where it is consumed in code and what constraints it implies:
     - Use tools: `bash("grep -rn '<knob_name>' . --include='*.py' | head -50")` then `file_viewer(...)`.
   - If you cannot find the knob being consumed, you MUST NOT include it in the plan.
4. **Mandatory cheap preflight checks (fail fast before training).**
   - Every experiment MUST include a cheap preflight BEFORE training:
     - config loader can parse the config, AND
     - model can be constructed (construction only; no training loop).
   - Prefer implementing this preflight inside the SAME task's command (so you don't need separate "prepare_config" tasks).
   - You MUST discover the correct import paths by reading code (do NOT assume module names).
5. **Dimensional/shape parameter discipline (for rank/dim/channels/heads/etc.).**
   - If a parameter name suggests a dimension, rank, channels, heads, layers, or feature count, you MUST explicitly reconcile it with the problem dimensionality (2D vs 3D) and the consuming module's expected shapes.
6. **Run from project root.**
7. **Result directory conventions:**
   - `result_dir` MUST be the step root: `_science_runs/step_{{step4}}/`
   - multiple tasks share the same `result_dir`; avoid collisions by using `{{task_id}}` in *filenames* (do not create subdirectories)
8. **No `/tmp` configs/data.** Never reference `/tmp/*.yaml` (HPC non-durable).
9. **Prefer single-task execution.**
   - Put config/data preparation inside the SAME `command` (use `bash -lc '...'` to chain steps).
   - Only introduce separate prep tasks when multiple downstream tasks reuse the exact same prepared artifact.
10. **Commands must be executable as-is.** The runner will not inject flags like `--output_dir`.
   - You may use `$SCIENCE_RESULT_DIR` (absolute path) or `{{result_dir}}` (preferred).
11. **No hard-coded step paths** inside commands. Always use `{{result_dir}}` or `$SCIENCE_RESULT_DIR`.
12. **Execution tasks must have metrics.** `metric_specs` must be non-empty for train/eval; if needed, add `collect_metrics_*`.

- `jsonl_last`: filter JSONL rows by `where`, then take the last row’s key
