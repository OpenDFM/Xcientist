Analyze the results of a series of experiments and draw scientific conclusions.

**HARD CONSTRAINTS (MUST FOLLOW):**
1. **NO CODE-LAYER FEEDBACK**: Do NOT generate any code optimization tickets. Always set `"optimization_tickets": []`.
2. **NO "RE-RUN" LANGUAGE**: Do NOT suggest "retry", "re-run", "rerun", or "run again". You may propose a *new* corrected plan in `next_experiments`.
3. **EVIDENCE-DRIVEN**: When proposing changes to the experiment plan, justify them using concrete evidence from stderr/stdout snippets and/or files under each task's `result_dir` (e.g., `meta_*.json`, `stderr_*.txt`, `stdout_*.txt`, `metrics_*.json`).
4. **PLAN MUST BE EXECUTABLE**: The proposed `next_experiments` must be runnable by the Experiment Manager/Worker.
   - Each task MUST have a unique `task_id`, a clear `description`, and an executable shell `command`.
   - Prefer writing outputs under `$SCIENCE_RESULT_DIR` (the Worker sets this env var).
   - Avoid writing to `/tmp/` (the Manager rejects such plans).
   - If you use placeholders, use ONLY: `{step}`, `{step4}`, `{task_id}`, `{plan_id}`, `{result_dir}`.
